# 第一部分:全流程技术介绍

## 0. 三分钟自述(开场版本)

"我在快手公开数据集 KuaiRand-1K 上独立完成了一个完整的两级推荐系统。数据是 1000 个用户一个月内的 1171 万条真实曝光日志,用户行为序列中位数 8300 条,是公开数据里少数能真正研究超长期序列建模的。项目分三层:召回层做了热门、ItemCF 和带 logQ 修正的双塔;排序层从 DIN 出发,实现了 SIM 式两阶段长序列检索,经过三次迭代把检索粒度做到二级类目;再把 437 万视频的 caption 用中文句向量模型编码成冻结语义表注入排序模型。最终点击 GAUC 从 0.577 提到 0.598,其中语义增强贡献 +1.7pt,且在冷启动 item 上增益放大到 +1.6pt AUC、呈单调梯度;这个结论我用数据集自带的随机曝光流量做了无偏验证。另外做了 MMoE 三目标预估和融合权重的 tradeoff 分析,以及一个 next-item 预训练的负结果分析。整个项目跑在一张 4060 上,每个涨点有消融,每个失败有定位。"

## 1. 数据与 ID 体系设计

**数据集**:KuaiRand-1K(CIKM'22),快手真实推荐日志。1,000 用户 / 4,369,953 个视频 / 11,713,045 条标准曝光(2022.04.08–05.08)+ 43,028 条随机干预曝光。12 种反馈信号(is_click、long_view、is_like、is_follow、play_time_ms 等)。用户序列长度:中位数 8,327,P90 24,418,最长 127,557——"超长期序列"的叙事由数据支撑而非包装。

**核心矛盾**:437 万 item、只有 1000 个用户,85.37% 的交互落在曝光 <30 次的长尾视频上。这逼出了三层 ID 体系(工业稀疏建模的缩影):

- `iid`(召回用):曝光 ≥30 次的 31,699 个视频编为 1..N,其余归 0。召回只在此候选集内检索——437 万全量 ANN 既无必要(低置信)也不经济。
- `iid_h`(排序/序列用):候选集保留真实 ID,其余 85% 长尾按 `video_id % 200000` 哈希进 20 万个桶,词表总大小 231,700。哈希冲突可接受:平均每桶约 21 个长尾视频,它们本来就没有足够信号支撑独立 embedding,分桶把"完全无表征"升级成"共享粗粒度表征"。作者 ID 同理哈希进 5 万桶。
- `vid`(语义查表用):全量视频的稠密索引 1..437 万。哈希会丢失视频身份,而语义表必须按视频精确查表,所以序列和样本同时携带 `iid_h` 与 `vid` 两套 ID——可训练 embedding 走哈希,冻结语义走稠密索引。

**切分协议**:严格按时间,训练 ≤05.01(885 万曝光)、验证 05.02–04(127 万)、测试 05.05–08(159 万)。随机切分在序列推荐里会把"未来"泄给"过去",是最常见的评估作弊。

**无泄漏的历史构建**:每条排序样本的用户历史通过 `searchsorted(times, t, side='left')` 严格截取 time < t,同毫秒事件也被排除(同毫秒可能是同一次曝光的点击回流,构成标签泄漏)。这一逻辑有专门单元测试覆盖(用户边界、时间边界、截断)。历史允许包含 val/test 期内、样本时刻之前的点击——线上 serving 时这些信息本来就可用,不属于泄漏。

## 2. 召回层

**基线阶梯**:热门(无个性化下界)→ ItemCF → 双塔。1K test:R@50 = 0.0021 / 0.0047 / 0.0100,R@500 = 0.0159 / 0.0296 / 0.0599。

**ItemCF 细节**:点击序列内滑窗(w=10)统计共现,对称化后做余弦归一 sim(i,j)=cooc/√(cnt_i·cnt_j)(惩罚热门 item 的虚假共现),每行只保留 Top-100 相似对控制内存与噪声;推断时对用户最近 50 个去重点击的相似向量求和,排除已点击。每用户参与共现统计的点击截断到最近 2000 条,控制 1K 上的 pair 数量。

**双塔**:用户塔 = 最近 50 次点击 embedding mean-pooling ⊕ 用户 ID embedding → MLP;物品塔 = item ⊕ 作者 embedding → MLP;两侧 L2 归一,in-batch softmax,温度 0.05。评估时全量 item 向量化后精确内积检索(3 万候选无需 ANN;规模上去换 faiss HNSW/IVF)。

**这里踩过的两个关键坑**(面试高频):

1. **in-batch 负采样的流行度偏差**。batch 内其他样本的正例被当作负例,而热门 item 出现在 batch 里的概率正比于其点击频率——热门 item 被系统性地过度惩罚。修正:sampled softmax 的标准校正,logits 减去 log Q(i)(i 为该 item 的采样概率,即训练点击占比)。效果:Pure 数据集 val R@50 从 0.07 → 0.267,近 4 倍——这不是调参,是修 bug 级别的提升。
2. **末轮保存 vs 最优保存**。双塔在 Pure 上 ep2 见顶后持续过拟合,保存末轮模型使 test R@50 損失 15%(0.199 vs 0.230)。加最优 checkpoint + patience=2 早停后解决。1K 上双塔训到 ep8 才饱和——召回和 CTR 模型的训练动力学完全不同(见 §6)。

## 3. 排序层:从 DIN 到细粒度 SIM

**DIN(短序列基线)**:target attention——历史每个 item 的表征与 target 拼接 [h, t, h⊙t, h−t] 过小 MLP 出权重,softmax 加权池化,再与 target、交叉项、用户 embedding 拼接过 MLP 输出 logit。BCE 训练,负样本即曝光未点击(真实负反馈,不需要采样)。hist_len=200,emb 64,MLP 256-128-64。结果:AUC 0.7159 / GAUC 0.5769。

**SIM 三次迭代**(项目最重要的问题定位链路):

- **v1:同类目最近 K=50 条检索** → 与 DIN 指标逐位相同(GAUC 0.5747 vs 0.5769)。诊断:用户日均点击约 280 条,短支路 200 条只覆盖最近约 17 小时,而"同类目最近 50 条"对高频类目几乎全落在这个窗口内——长短支路喂的是同一批行为,attention 学不出增量。
- **v2:不相交检索** → 长支路只检索**短窗口起点时间之前**的同类目行为(实现:`window()` 返回窗口起点时间戳 t_start,GSU 以 t_start 为右边界检索),按构造零重叠。val +0.3pt,test 持平(0.5775)——远期信息有增量但被检索粒度和长尾表征质量卡住。
- **v3:检索键升级为二级类目**(cat2,175 类,来自官方 categories 补充文件)→ test GAUC **0.5976**,较 v2 +0.3pt。66 个一级标签下"同类目"太宽泛(比如"影视"),175 个二级类目("电视剧/影视解说/…")下检索出的行为与 target 的相关性质变。

**序列长度 scaling 消融**(SIM+sem):hist_len 50/100/200/400 → GAUC 0.5857 / 0.5894 / 0.5947 / 0.5946。单调上升、200 饱和——这条曲线本身就是"为什么需要检索式长序列而不是暴力加窗"的论据:注意力窗口的边际收益归零,但全生命周期里仍有可检索的增量(v3 的 +0.3pt 就在窗口之外)。

## 4. 语义增强(主要涨点)

**动机**:v2 的诊断把瓶颈指向长尾表征——检索出的远期行为大多是哈希桶 ID,embedding 冲突噪声大。协同信号在长尾处天然稀疏,需要不依赖点击量的表征来源,即内容语义。

**管道**:bge-small-zh-v1.5 编码 caption+封面文本(L2 归一,512 维)→ 覆盖 3,748,933 个视频(85.79%,缺失置零)→ IncrementalPCA 降到 64 维(保留 54.6% 方差)→ fp16 语义表(437 万 × 64,560MB,常驻 GPU)→ 一层可训练 Linear(64→64) 投影后,**加**到 target 和长短序列每个 token 的 ID 表征上。语义表冻结:65 万训练点击 fine-tune 一个语言模型只会灾难性遗忘,冻结表 + 可训练投影是小数据下注入世界知识的正确姿势。

**结果(2×2 消融,1K test GAUC/AUC)**:DIN 0.5769/0.7159 → DIN+sem 0.5943/0.7291;SIM 0.5775/0.7148 → SIM+sem 0.5947/0.7299。语义贡献 +1.7pt GAUC,是全项目最大单项。

**分桶验证(语义故事的点睛)**:按 item 训练期曝光量分桶看 ±sem 的 AUC 差:0 次(纯新)+1.64pt、1–9 次 +1.40、10–29 +0.82、30–99 +0.36、100+ +0.51——增益随冷度单调放大,直接证明"世界知识补足协同信号缺口"。100+ 桶仍有 +0.5pt:热门 item 的语义帮助的是"用户侧匹配"而非"item 侧表征"。

**无偏验证**:test 期随机曝光(等概率展示,无策略偏差)上,sem 0.6876/0.5863 vs nosem 0.6760/0.5691——增益成立;同时标准日志 AUC 虚高约 4pt,量化了曝光偏差。

## 5. 多任务(MMoE)

SIM+语义的表征层为共享底座 → 4 个专家 MLP(输入 384 维,输出 64)→ click/long_view/like 各自 softmax 门控加权专家输出 → 各自任务塔出 logit,等权 BCE。结果(test):click 0.7302/0.5939(与单任务 0.5947 打平,无损)、long_view 0.7419/0.6152、like 0.9180/0.5961。

融合权重扫描(tradeoff 工具):score=(1−α)p_click+α p_lv,α 从 0→0.4 时点击命中几乎无损(−0.06pt)而长观命中 +2.1pt(免费午餐区),α→1.0 进入 1pt 点击换 3.5pt 长观的真实取舍区;同时 top-10 类目多样性 4.94→4.69 单调收窄。

## 6. 训练动力学与工程

**One-epoch 现象**:DIN/SIM/MMoE 的 val GAUC 全部在第 1 个 epoch 见顶(如 DIN/1k:ep0 0.5825 → ep1 0.5731),复现了工业界 CTR 模型"只训一遍"的经典结论——大规模稀疏 ID embedding 第二遍就开始记忆样本。而双塔召回需要 8+ epoch:它的监督信号(采样 softmax)每个 epoch 看到的负例组合不同,等效数据量更大。统一用"val 指标 + patience=2 早停"处理两种动力学。

**预训练负结果**:2 层 Transformer(d=64, 2 heads, causal 目标为 next-item,in-batch softmax + logQ)在 64.3 万点击事件上预训练 item embedding,迁移到 SIM 微调:GAUC 0.5945 ≈ 基线 0.5947。原因:语料(64 万事件)相对词表(23 万)严重不足,预训练 loss 停在 4.8 远未收敛;且预训练信号与下游同源(都是点击流),信息冗余。适用条件推断:跨域/异源信号 + 更大语料。

**资源账**(单张 4060, 8GB):最大参数块是 iid_h embedding 表 231,700×64 ≈ 1500 万参数;语义表 560MB fp16 buffer;排序一个 epoch(885 万样本, batch 1024)约 8–11 分钟,瓶颈在 DataLoader 的 per-sample searchsorted(4 workers 缓解);全项目所有实验总 GPU 时长约 20 小时。

# 第二部分:拷打问题与参考回答(46 题)

## A. 数据与评估(最常被问)

**Q1:为什么用 GAUC 而不是 AUC?怎么实现的?**
A:全局 AUC 混入了"区分活跃/不活跃用户"的信号——把活跃用户(点击率高)的正例排在不活跃用户的负例前面就能刷高 AUC,但这对"给某个用户排序"毫无意义。GAUC 按用户分组算 AUC 再按该用户曝光数加权平均,跳过全正/全负用户,直接对应线上排序体验。我的实现里两者差距约 13pt(0.73 vs 0.60),差值就是被 AUC 高估的部分。

**Q2:为什么时间切分?随机切分或 leave-one-out 有什么问题?**
A:随机切分会让训练集包含测试时刻之后的行为,模型能"看到未来"——比如通过共现学到测试期才流行的内容。leave-one-out(每用户最后一条做测试)对序列模型稍好,但不同用户的"最后一条"时间不同,与线上"某时刻全量切换新模型"的场景不符,还会低估分布漂移。时间切分是唯一与线上一致的协议,我的 val/test 各覆盖 3–4 天。

**Q3:如何保证特征无泄漏?细到什么程度?**
A:三层防护:(1) 历史序列 searchsorted 严格 time < t,同毫秒排除——同毫秒事件可能是同一次曝光的点击回流;(2) 数据集自带的视频统计特征文件是全月均值,含未来信息,整体弃用;(3) 负采样(双塔)只从训练期采。历史截取逻辑有单元测试:用户边界不串、时间边界严格、K 截断取最近。

**Q4:负样本怎么构造的?**
A:排序和召回不同。排序用曝光未点击作负例——这是真实负反馈,是 CTR 建模的标准做法;召回训练用 in-batch 负采样(等效从点击分布采负例),因为召回要面对全库、不能只学"曝光内区分"。这也引出两者样本空间不一致的 SSB 问题(见 Q18)。

**Q5:85% 交互是长尾 OOV,你怎么处理?哈希冲突不会有问题吗?**
A:分层处理:召回候选集只留 31,699 个高频 item(全量检索低频 item 置信度不够);排序和序列侧把长尾哈希进 20 万桶。冲突平均 21 个视频共享一个桶,但这些视频人均曝光 <30 次,本来就撑不起独立 embedding——分桶把"无表征"变成"共享粗表征",再用语义特征(按 vid 精确查表,无冲突)补足个体性。消融证明这个组合有效:语义增益恰好在长尾桶最大。

**Q6:桶数怎么定的?**
A:量级权衡:桶数太少冲突严重(每桶上百视频,表征糊成均值),太多则每桶样本不足、且 embedding 表膨胀。20 万桶对 437 万长尾视频是 1:21,每桶在训练集中期望有几十次曝光,够学一个粗粒度表征;embedding 表 231,700×64 约 1500 万参数,8GB 显存舒适。没有精细调过,这类超参的合理区间很宽。

**Q7:无偏评估具体怎么做?为什么随机曝光能去偏?**
A:标准日志的候选是旧策略筛过的——模型学的是"在旧策略认为好的东西里再排序",离线 AUC 可能只是"更像旧策略"。KuaiRand 的随机干预流量把每条展示以等概率替换成随机视频,该子集上的评估不依赖旧策略偏好。我只用 test 期的随机曝光(4.3 万条,避免与训练期重叠):AUC 从 0.730 掉到 0.688(偏差约 4pt),但 ±语义的差距保持(+1.2pt AUC)——收益是真的。

**Q8:1000 个用户会带来什么问题?**
A:三个:GAUC 是 1000 个用户级 AUC 的加权平均,方差偏大,小差异(<0.3pt)要谨慎解读;用户 embedding 极易记忆(也是 one-epoch 的推手之一);无法研究用户冷启动。我的应对是所有关键结论都配多角度验证(分桶、无偏、消融同向)。

**Q9:val/test 样本的历史包含 val/test 期点击,不算泄漏吗?**
A:不算。线上 serving 某时刻 t 的请求,用户在 t 之前的所有行为(包括今天的)都是可用的。泄漏的定义是使用 t 时刻及之后的信息。我泄漏防护的对象是标签(严格 <t,同毫秒排除)而不是"训练/测试期"这个人为边界。

**Q10:is_click 这个标签的实际含义?**
A:KuaiRand 的定义在双列 UI 是点击,单列 UI(主 feed)是 valid_play:短视频看完、长视频看超 7 秒。所以它更接近"有效消费"而非字面点击,正例率 37.8%。这也是我多任务里再加 long_view(18 秒口径)的原因——两个时长阈值刻画不同深度的兴趣。

## B. 召回

**Q11:双塔为什么 L2 归一 + 温度?**
A:归一后内积=余弦,把打分范围固定到 [-1,1],训练稳定且与线上 ANN(余弦/内积检索)一致。温度 0.05 把 logits 放大 20 倍再进 softmax:余弦差异天然很小,不放大则 softmax 接近均匀、梯度弱;温度本质控制对困难负例的关注程度,是对比学习的标准组件。

**Q12:logQ 修正的推导讲一下?**
A:我们想优化全库 softmax P(i|u)=exp(s_ui)/Σ_j exp(s_uj),但只能在 batch 内算。batch 内负例 j 以概率 Q(j)(∝点击频率)被采到,importance correction 下等效 logits 应为 s_uj − log Q(j):热门 item 因常被当负例而多扣的分,由 −logQ(大的 Q 扣得多)补回来。实现上就是交叉熵前对每列减 log(该 item 训练点击占比)。效果 Pure val R@50 0.07→0.267。

**Q13:为什么双塔一开始不如 ItemCF,后来在 1K 上反超一倍?**
A:一开始输是因为两个 bug 级问题(logQ 缺失 + 末轮保存)。修复后在 Pure(7.5k 候选)上与 ItemCF 相当——小候选集里记忆式共现已经够用;1K(3.2 万候选)上双塔 R@50 翻倍——候选空间越大,泛化式的稠密检索相对记忆式统计的优势越明显,ItemCF 的共现矩阵在大空间里急剧稀疏。

**Q14:ItemCF 为什么用 sqrt 归一?**
A:裸共现计数会被热门 item 支配(和谁都共现)。除以 √(cnt_i·cnt_j) 是余弦归一,把"共现次数"变成"共现倾向"。这与双塔的 logQ 是同一个思想在不同方法里的体现——去流行度偏差是召回的永恒主题,面试我会主动把这两处联系起来讲。

**Q15:召回评估为什么把训练期已点击的 item 从 ground truth 里去掉?**
A:评估目标是"发现新兴趣"的能力。已点击 item 的召回是"复读",指标会被刷高但没有产品价值(短视频场景重复消费很少)。推断时也同步排除已点击。

**Q16:双塔和 CTR 模型为什么 epoch 行为不同?**
A:CTR 的样本-标签对是固定的,稀疏 ID 第二遍开始记忆;双塔的有效训练信号是 (user, pos, 采样负例集),负例组合每个 epoch 都在变,等效样本空间大得多,所以 1K 上到 ep8 才饱和。可以说 in-batch 采样自带数据增广。

**Q17:线上部署召回怎么做?**
A:物品塔离线全量推理建 ANN 索引(этот规模 faiss HNSW 即可,亿级用 IVF-PQ 有损压缩换内存);用户塔在线实时推理(毫秒级);索引按天/小时增量更新,新 item 靠语义侧特征进物品塔解决冷启动入库。双塔结构的全部意义就在于内积可 ANN 化。

**Q18:排序模型训练在曝光样本上,但要给召回结果打分,样本选择偏差(SSB)怎么办?**
A:诚实回答:本项目未处理,评估也在曝光空间内。工业做法:精排训练时混入少量召回未曝光样本作 easy negative、或用曝光概率做逆倾向加权(IPS);我的随机曝光数据其实提供了无偏样本源,可以作为 IPS 的校准集,这是我列的后续方向。

## C. 排序与长序列

**Q19:target attention 为什么优于 mean-pooling?attention 输入为什么拼 h⊙t 和 h−t?**
A:用户兴趣是多峰的,mean-pooling 把游戏、美食、剧集糊成一个均值向量;target attention 让"当前候选是游戏"时只激活历史里的游戏行为,等于按需读取的软检索。h⊙t、h−t 是显式交互特征,让打分 MLP 不必自己学出乘性/差性关系,小参数量下收敛更快——这是 DIN 原文的设计,我保留并能解释。

**Q20:你用 softmax 归一,DIN 原文特意不归一,为什么?**
A:原文不归一是想保留兴趣强度(历史里游戏行为多,总激活就该大)。我用 softmax 是工程取舍:训练更稳定,且强度信息我通过"长期类目点击次数"路径(GSU 检索数量、类目 embedding)部分保留。两种做法在我的数据规模上差异应在噪声内,但被问到必须知道原文的设计动机。

**Q21:SIM v1 和 DIN 指标几乎逐位相同,你怎么定位问题的?**
A:先排除实现 bug(单测确认检索正确),再做量化推理:日均 280 条点击 × 短窗口 200 条 ≈ 17 小时覆盖,而高频类目的"最近 50 条同类目"时间跨度约 1–2 天,大部分落在窗口内——两支路输入高度重叠,attention 从冗余信息里学不出增量。验证方式是 v2 改成不相交检索后 val 出现 +0.3pt,证实了重叠假设。这题我会主动讲,它展示的是"指标不动时的诊断方法论"。

**Q22:不相交检索怎么实现的?**
A:短支路取窗口时返回窗口内最早一条的时间戳 t_start,长支路检索的右边界从样本时刻 t 换成 t_start——两支路在时间轴上天然不相交,零额外成本。比起"检索后去重"的实现,按构造不相交更干净且可单测。

**Q23:cat2 为什么有效?继续用三级类目会更好吗?**
A:检索键的粒度决定"相关"的定义:66 类下"影视"太宽,检索出的远期行为一半不相关;175 类下"影视解说 vs 电视剧"分开,信噪比质变,+0.3pt。三级类目(千级)会走向另一个极端——每个键下行为太少,长支路经常空转,且新类目泛化差。最优粒度取决于行为密度,原则上可以按"每键平均行为数"选择,极限形态就是 embedding 软检索(连续相关性,无需离散键)。

**Q24:hist_len 到 400 为什么饱和?**
A:两个原因:mean 上更远的行为与当前兴趣相关性衰减,attention 虽能选择但 padding 稀释和噪声行为增多;且 400 条也只覆盖约 1.5 天,和 30 天的全量历史比仍是"近期"——真正的远期增量要靠检索(v3 证明了窗口外确实有 +0.3pt 可拿)。这条饱和曲线是"检索式长序列"的最有力论据。

**Q25:embedding 软检索(soft search)怎么实现?复杂度?**
A:ETA/TWIN 路线:给全历史行为离线建向量(我的语义表现成可用),线上对 target 向量做 top-K 内积检索。朴素实现每样本 O(历史长度×d),1 万条历史不可行;工业做法是 ETA 的 SimHash(把内积近似成汉明距离,位运算)或 TWIN 的 GPU 并行精确检索 + 缓存。我的下一步计划就是用语义向量做 GSU,复用已有资产。

**Q26:长序列模块线上延迟怎么控制?**
A:关键是把检索从"请求时扫全史"变成"预建索引查表"。我的 TagHistory 本质就是 (user, cat) → 时间有序行为列表的倒排索引,线上对应用户行为的按类目分桶存储,检索是 O(log n) 的二分;SIM 原文同样是离线建好的两级索引。增量行为追加写,索引不需要重建。

**Q27:GSU 为什么用 target 的类目检索,而不是用户最常点的类目?**
A:检索必须是 target-aware 的:排序打分回答"这个候选和用户的相关性",用用户 top 类目检索的话,所有候选共享同一份长期上下文,失去区分度,退化成静态用户画像。target 类目检索让每个候选激活与它相关的那段历史,和 target attention 的逻辑一脉相承。

**Q28:one-epoch 现象的机制?怎么应对?**
A:机制上是稀疏 ID embedding 的记忆:每个 ID 的参数只被它自己的样本更新,第一遍学到泛化模式,第二遍开始拟合样本噪声(尤其低频 ID),val 立刻掉头。应对:一遍数据 + 早停是主流;也有工作区分 embedding 层和 MLP 层用不同学习率/epoch。我的数据上 DIN/SIM/MMoE 全部 ep0 见顶,双塔例外(负采样自带增广),这个对比本身是个好答案。

## D. 语义增强

**Q29:为什么选 bge-small-zh?为什么冻结不微调?**
A:选型三因素:中文语料训练(caption 是中文短文本+话题标签)、512 维小模型够用且编码快(437 万条在 4060 上约 40 分钟)、检索任务预训练与"相似内容相近表征"的用途一致。冻结:65 万点击样本 fine-tune 1 亿参数编码器必然灾难性遗忘+过拟合;冻结表+可训练投影(64×64)让梯度只调整"语义空间到行为空间的映射",保留世界知识。这是小数据接大模型的标准姿势。

**Q30:PCA 到 64 维丢了一半方差,不心疼吗?**
A:权衡显存:512 维 fp16 表要 4.5GB,超过 4060 的余量;64 维 560MB 可常驻。54.6% 方差听着低,但 PCA 方差和下游效用不是线性关系——语义空间的主方向(话题、品类)集中在头部维度,尾部很多是表述风格噪声。事后验证:64 维已带来 +1.7pt,是全项目最大单项;维度消融(128/256)列为后续,预期边际收益递减。

**Q31:语义为什么用加法融合,不用 concat 或 gating?**
A:加法把语义投影到与 ID embedding 同一空间再叠加,参数最少、不改变下游所有结构的输入维度(attention、MLP 全部复用),且允许 ID 表征在语义先验上做残差学习——行为信号充足的 item,ID embedding 可以覆盖语义;冷 item 则退化到纯语义。concat 会让序列 token 维度翻倍,attention 参数跟着翻倍,小数据上更容易过拟合。gating 是合理的升级方向,但应该在加法基线证明价值后再加复杂度。

**Q32:15% 视频没有 caption,零向量会不会伤害模型?**
A:零向量经过线性投影后是常数偏置,等效于"无语义信息"的默认表征,模型可以学到把它当成一种缺失标记。更优做法是给缺失单独一个可学习向量、或用类目名文本兜底编码。分桶实验里未见缺失导致的异常,但这是我知道的待优化点。

**Q33:凭什么说语义增益来自"世界知识"而不是单纯多了一路特征?**
A:分桶实验就是为回答这个设计的:如果只是"多一路特征",增益应该在各热度桶均匀;实际增益随 item 冷度单调放大(纯新 +1.64pt vs 热门 +0.51pt),说明它替代的正是行为信号缺失的部分——纯新 item 没有任何协同信息,+1.64pt 全部来自内容理解。且无偏评估排除了"更像旧策略"的解释。

**Q34:了解语义 ID / TIGER 吗?和你的方案什么关系?**
A:TIGER 用 RQ-VAE 把内容向量量化成离散语义 ID 序列,item 由语义 ID 表示,天然解决冷启动且支持生成式检索(自回归生成语义 ID)。我的方案是"连续语义向量做特征",TIGER 是"离散语义编码做身份";前者改动小、即插即用,后者是范式级改变(配合生成式召回)。我的语义表其实是 TIGER 的上游资产——加一步 RQ-VAE 量化就能实验语义 ID,这是我 roadmap 里的一项。

**Q35:多模态怎么扩展?**
A:架构上是零改动的:视频封面/帧过视觉编码器(如 Chinese-CLIP)得到向量,与文本向量拼接或平均后进同一个 PCA+冻结表+投影管道。真正的问题是数据集不提供原始图像/视频,所以我只做了文本模态,视觉留作接口。如果两种模态都有,值得做对齐(CLIP式对比)而非简单拼接。

## E. 多任务

**Q36:MMoE、shared-bottom、PLE 的区别和取舍?**
A:shared-bottom 所有任务共享一个底座,任务冲突时互相拖累(负迁移);MMoE 用多专家+每任务门控,让任务软性选择专家组合,缓解冲突;PLE 更进一步区分共享专家和任务私有专家,分层提取。我选 MMoE 是复杂度和收益的平衡点:3 任务 4 专家已验证 click 无损 + 两个副任务可用;PLE 是 like 这种稀疏任务需要更强隔离时的升级项。

**Q37:任务冲突你观测到了吗?loss 为什么等权?**
A:观测:单任务 click GAUC 0.5947,MMoE click 0.5939——差异在噪声内,说明 4 专家的容量下三任务基本不冲突(long_view 与 click 本来相关性就高)。等权是基线选择:三个 BCE 量纲相同;like 正例率 1.5%,其 loss 绝对值小,等权下实际已经是弱任务。若要精细,可用 uncertainty weighting 或 GradNorm,但应先有"等权不行"的证据。

**Q38:like 的 AUC 0.918 为什么和 GAUC 0.596 差那么远?**
A:这是 AUC 被用户间差异撑大的极端例子:like 是强用户属性行为(有人爱点赞有人从不点),模型识别"谁是点赞型用户"就能把全局 AUC 推很高;但同一用户内部哪条内容会被赞(GAUC)难得多。这组数字是我"为什么必须看 GAUC"的最佳例证,面试我会主动引用。

**Q39:线上融合权重怎么定?**
A:离线先用权重扫描画 Pareto 曲线定候选区间(我的曲线显示 α∈[0,0.4] 是免费午餐区),再小流量 AB 验证线上真实指标(时长、留存),权重最终由业务目标拍板——算法交付的是曲线而非单点。更自动化的方向是把融合做成约束优化(最大化时长 s.t. 点击不降超 x%)。

## F. 工程与训练

**Q40:显存怎么分配的?最大的头是什么?**
A:语义表 560MB(fp16 buffer)+ iid_h embedding 表 1500 万参数×4B×3(参数+Adam 两个矩)≈ 180MB + 激活值(batch 1024 × 250 tokens × 64 维的序列 embedding 及 attention 中间量,数百 MB)。总占用约 2–3GB,8GB 从容。若上 512 维语义或 emb_dim 128 就要开始精打细算,可选 fp16 训练或梯度检查点。

**Q41:训练瓶颈在哪?**
A:数据侧。每个样本要做 2 次 searchsorted(短窗口+GSU)+ numpy 切片拷贝,GPU 前向反向反而不满载。缓解:4 个 DataLoader worker 并行、历史索引全部预构建成连续 numpy 数组(避免 pandas 逐行)、fp16 语义表查表在 GPU 侧做。进一步优化是把样本历史离线物化成定长数组存 parquet,训练时零计算,代价是磁盘×序列长度。

**Q42:超参怎么调的?哪些敏感?**
A:原则是"只调有理论依据的":温度(0.05,对比学习常规值)、logQ(不是超参是修正项)、hist_len(专门做了 scaling 消融)、long_topk 和桶数(量级推理,未细调)。lr 固定 1e-3 Adam,batch 由显存决定。最敏感的是温度和 logQ——它们是"能不能 work"级别;其余是"好一点差一点"级别。有限算力下把预算花在消融而不是网格搜索,这是我的取舍。

**Q43:为什么早停 patience=2 而不是 1?**
A:val 指标有噪声(尤其 GAUC 在 1000 用户上),单次下降可能是波动;patience=2 容忍一次假回撤。配合"保存最优而非最后",早停时机只影响训练时长不影响最终模型。

**Q44:代码质量怎么保证的?**
A:三层:关键正确性(历史截取无泄漏、GSU 时间/类目隔离、ID 范围)写了单元测试;全流程先在合成数据(复刻真实 schema)上冒烟再跑真实数据;所有实验配置化(yaml + 命令行覆盖),结果统一追加到 results.csv,任何数字可溯源到确切配置。

## G. 前沿延伸(考察视野)

**Q45:这个系统怎么演进成 LRM / 生成式推荐(HSTU、OneRec)?**
A:我的架构已具备三个前置件:统一 token 化(item/作者/类目/语义共入一个表征空间)、长序列处理、语义 ID 的上游资产(内容向量)。演进路径:(1) 把"target attention 打分"换成"自回归 Transformer 编码全行为流",训练目标从 pointwise CTR 变成 next-token(HSTU 的做法,它还证明了推荐里存在 scaling law);(2) 把 item 表示换成 RQ-VAE 语义 ID,召回变成生成式(TIGER/OneRec);(3) 排序和召回合一,端到端生成推荐列表。当前挡在个人项目前面的是算力和数据量——HSTU 的收益在万亿参数、千亿样本级才显现,我的 hist_len 饱和曲线其实就是小数据下 scaling 受限的一个切片。

**Q46:sparse MoE 和 RL 在这个系统里的位置?**
A:MoE:我的 MMoE 门控是软路由(所有专家都算),用户/场景异质性大时改 Top-k 稀疏路由,容量随专家数扩、计算不随之扩——这是承接"1000 用户→数亿用户"异质性的标准路。RL:当前是单步 CTR 视角,没有建模推荐对用户状态的改变(茧房就是负外部性);做 RL 需要无偏的 off-policy 评估,我的随机曝光数据恰好是 OPE 的黄金标准数据源——可以先做 IPS/DR 的策略评估,再考虑保守的离线 RL(如 CQL)。我会强调:RL 的前置件是可靠的 reward 与评估,盲目上 RL 是本末倒置。
