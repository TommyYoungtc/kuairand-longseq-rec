## 一、项目总览

**一句话定位**:在快手公开数据集 KuaiRand-1K 上,独立完成"召回 + 精排"两级推荐系统,核心探索三个方向——超长期用户序列建模(SIM 式两阶段检索,三次迭代)、大规模稀疏特征建模(三层 ID 体系 + 长尾哈希分桶)、语义/世界知识增强(437 万视频 caption 的冻结语义表)——并配套无偏评估、冷启动分桶、序列长度 scaling 等验证实验。

**最终成绩单**(1K test,点击任务):

| 模型 | GAUC | AUC | 增量归因 |
|---|---|---|---|
| DIN(短序列 200,纯 ID) | 0.5769 | 0.7159 | baseline |
| SIM v2(+不相交长序列,一级类目) | 0.5775 | 0.7148 | +0.06pt |
| DIN + 语义 | 0.5943 | 0.7291 | 语义 +1.74pt |
| SIM + 语义(一级类目) | 0.5947 | 0.7299 | — |
| SIM + 语义 + 二级类目 GSU | 0.5976 | 0.7316 | 细粒度检索 +0.29pt |
| **v4:+ 用户内 pairwise 损失** | **0.6054** | **0.7356** | 目标-指标对齐 +0.78pt |

v4 说明:GAUC 是用户内排序指标而 BCE 优化全局校准——目标与指标不对齐。自定义用户分组 batch(128 组 × 每组 8 条同用户曝光)+ 组内 BPR 辅助损失(权重 0.5)。消融:关 pairwise 掉回 0.5979(几乎全部涨幅来自它);同批加入的用户画像特征零增益——1000 用户下 user_id embedding 已记忆一切,画像冗余,其价值在 ID 记不过来的亿级规模才显现(这个"为什么没用"的解释本身是高分答案)。

辅助结论:语义增益在纯新 item 上 +1.64pt AUC 且随冷度单调;无偏评估(随机曝光)确认收益真实并量化出标准日志约 4pt 的 AUC 虚高;序列窗口暴力加长在 200 处饱和;MMoE 三任务无损扩展;next-item 预训练为可解释的负结果。全部实验跑在单张 RTX 4060(8GB)上,总 GPU 时长约 20 小时。

**叙事主线**(面试讲述顺序):数据的结构性矛盾(超长序列 + 85% 长尾)→ 三层 ID 体系应对稀疏 → 召回层踩坑与修复(logQ、checkpoint)→ 排序从 DIN 到 SIM 的三次迭代(方法论高光)→ 诊断指向表征质量 → 语义增强(最大涨点 + 双重验证)→ 多任务与权衡 → 负结果与边界。这是一条"发现问题—定位问题—解决问题—验证结论"的完整闭环,比单纯罗列模型有说服力得多。

## 二、数据集深度解析

### 2.1 KuaiRand 是什么、为什么选它

KuaiRand(CIKM'22)是快手发布的推荐日志数据集,其独特之处是包含**随机干预流量**:在标准推荐流里,每条展示以固定概率被替换为从池子里等概率抽取的随机视频(全量数据中约 0.37% 的展示被替换)。这使它成为极少数支持无偏评估 / off-policy evaluation 研究的公开数据集。官方提供三个版本:27K(全量,46GB)、1K(抽 1000 用户及其全部相关视频,4.3GB)、Pure(只保留随机池内 7583 个视频的日志,194MB)。

我的用法:**Pure 做管道调试**(小、快、schema 与 1K 完全一致),**1K 做正式实验**。选 1K 而非 Pure 做主战场的原因:Pure 的序列是不完整的(只保留池内视频的交互,用户序列中位数仅 39),而 1K 保留用户的完整行为流——序列长度中位数 8,327、P90 达 24,418、最长 127,557,"超长期序列建模"的叙事由数据支撑。2024 年官方追加了两个补充文件:`kuairand_video_captions.csv`(视频标题 + 封面文字,3.2GB)和 `kuairand_video_categories.csv`(四级层次类目,3.7GB),按 `final_video_id` 与主数据对齐——这两个文件分别支撑了我的语义增强和细粒度 GSU。

### 2.2 数据规模与信号

1K 版:1,000 用户 / 4,369,953 视频 / 11,713,045 条标准曝光(2022.04.08–05.08,31 天)+ 43,028 条随机干预曝光。每条日志 12 种反馈信号,我实际使用四种:

- `is_click`:双列 UI 下是点击;单列主 feed 下定义为 valid_play(时长 ≤7s 的视频看完、>7s 的看超 7 秒)。正例率 37.8%。注意它的真实语义是"有效消费"而非字面点击,这影响对指标的解读。
- `long_view`:更深的消费(≤18s 看完 / >18s 看超 18 秒),正例率 26.2%。
- `is_like` 正例率 1.56%、`is_follow` 0.10%——极稀疏信号,只适合做多任务副目标,不适合做主标签。
- `play_time_ms / duration_ms` 保留备用(完播率一类连续目标)。

item 侧特征使用 `video_features_basic`(作者 ID、时长、tag 列表、音乐等 62 个字段中的少数几个)。**特意弃用** `video_features_statistic`:它是每视频全月平均统计(展示数、播放数等),对任何训练期样本都包含未来信息,属于时间泄漏源——被问"你用了哪些特征"时,这个"没用什么、为什么"的回答比"用了什么"更展示功力。

### 2.3 EDA 中驱动设计的三个数字

- **序列长度中位数 8,327**:短窗口模型(如 hist=200)只覆盖日均消费(约 280 条)的最近 17 小时,超过 97% 的历史被丢弃——长序列方向立项依据。
- **候选裁剪后 OOV 85.37%**:曝光 ≥30 次的视频只有 31,699 个,覆盖 14.63% 的交互——三层 ID 体系与语义增强的立项依据。
- **每日曝光量稳定在 33–44 万**:切分点两侧数据量均匀,时间切分不会造成训练/评估失衡。

## 三、数据工程与 ID 体系

### 3.1 三层 ID 设计(应对 437 万 item × 1000 用户的极端稀疏)

**`iid`(召回专用)**:曝光 ≥30 的 31,699 个视频编号 1..N,其余归 0(OOV 桶)。召回只在候选集内检索,理由有二:长尾 item 的协同信号不足以支撑可信检索;全量 437 万的 ANN 索引在本项目规模下纯属浪费。阈值 30 的确定:EDA 里对覆盖率-候选数曲线的量级判断,保证候选集内 item 平均有足够曝光学出 embedding。

**`iid_h`(排序与序列专用)**:候选集保留独立 ID(1..31,699),其余 85% 长尾按 `n_cand + 1 + video_id % 200000` 哈希进 20 万个桶,词表 231,700。设计动机:排序模型的行为序列若沿用 `iid`,85% 的 token 会折叠成同一个 0 号,序列信息被摧毁;哈希分桶让长尾行为保留"粗粒度身份"。冲突率:平均每桶约 21 个长尾视频共享 embedding——这些视频人均曝光 <30,本就撑不起独立表征,分桶把"无表征"升级为"共享表征",且后续语义特征(按视频精确查表)补足了个体性。作者 ID(取值范围到 880 万)同理哈希进 5 万桶,公式 `author_id % 50000 + 1`,0 留给缺失。tag 取列表首个标签(0–65)加一平移,0 为缺失,词表 68 并做 clip 防越界。

**`vid`(语义查表专用)**:日志中出现过的全部视频稠密编号 1..4,369,953,行 0 保留为 padding。哈希丢失身份而语义表必须按视频精确索引,因此交互表、排序样本、行为序列全部同时携带 `iid_h` 与 `vid` 双 ID——可训练参数走哈希空间(控制词表),冻结语义走稠密空间(保留身份)。这个"双 ID 并行"设计是整个项目把稀疏 ID 与内容语义两个世界连接起来的枢纽。

二级类目 `cat2`:从 categories 补充文件按 `final_video_id` join,取 second_level_category_id,175 个有效取值,缺失/无效(-124)置 0。它只作为 GSU 检索键,不进 embedding。

### 3.2 切分与样本构建

时间切分:训练 ≤05.01(8,854,929 条曝光)、验证 05.02–05.04(1,271,580)、测试 05.05–05.08(1,586,536)。排序样本即曝光本身(正例=点击,负例=曝光未点击,无需负采样)。召回侧另建:`train_clicks`(训练期候选集内点击,647,699 条,993 个用户)作为双塔/ItemCF 的训练数据与序列来源;召回 ground truth 定义为"该评估期内点击的、且训练期未点击过的候选 item"(val 90,383 对 / test 99,377 对)——排除已点击是为了评估"发现新兴趣"而非"复读"。

### 3.3 泄漏防护(三道闸)

1. **历史严格截断**:每条排序样本的用户历史用 `np.searchsorted(times, t, side='left')` 截取 time < t,`side='left'` 保证同毫秒事件也被排除——同毫秒记录可能是同一次曝光的点击回流,构成标签泄漏。
2. **历史允许跨切分边界**:val/test 样本的历史包含该样本时刻之前的一切点击(含 val/test 期内的)——线上 serving 时这些信息本可用,这不是泄漏;泄漏的定义是"使用 t 时刻及之后的信息",防的是标签不是日历。
3. **单元测试固化**:`tests/test_rank_dataset.py`、`tests/test_sim.py` 用手工构造的微型点击流断言:用户边界不串、时间严格性、K/L 截断取最近、类目隔离、双 ID 平行一致。全流程先在合成数据(复刻真实 schema 的生成器)上冒烟,再上真实数据。

## 四、评估体系

**GAUC**:按用户分组计算 AUC,以该用户曝光数加权平均,跳过标签全同(全正/全负)的用户。动机:全局 AUC 混入"区分活跃/不活跃用户"的信号——把高点击率用户的正例排到低点击率用户的负例之前即可刷高,但对"给某个人排序"毫无意义。本项目最有力的例证是 like 任务:AUC 0.918(识别"谁是点赞型用户"容易)而 GAUC 仅 0.596(判断"这个人赞哪条"难),32pt 的剪刀差全部是被全局 AUC 高估的假象。点击任务上两者差距约 13pt(0.73 vs 0.60)。

**召回指标**:Recall@K 与 NDCG@K(K=50/100/500),对有 ground truth 的用户求平均;推荐列表排除该用户训练期已点击 item 与 OOV 桶。

**无偏评估**:仅取 test 期(date > 05.04)的随机干预曝光复评已训模型。原理:标准日志的候选由旧策略筛选,存在曝光偏差——离线提升可能只是"更像旧策略";随机流量的展示与策略无关,其上的指标反映真实排序能力。结果:SIM+sem 在随机流量上 AUC 0.6876 / GAUC 0.5863,无语义版 0.6760 / 0.5691——两点结论:标准日志 AUC 虚高约 4pt(0.73 → 0.69);语义增益(+1.2pt AUC / +1.7pt GAUC)在无偏数据上完整保留,排除了偏差假象。

**实验管理**:所有配置 yaml 化,命令行 `--set key=value` 覆盖(如 `--set rank.model=din rank.hist_len=50`),每次实验以带后缀的名字追加写入 `reports/results.csv`,任何数字可反查到确切配置与时间戳。

## 五、召回层

### 5.1 基线:热门与 ItemCF

**热门**:训练期点击量排序,推荐时排除用户已点击。1K test R@50=0.0021——个性化价值的下界标尺。

**ItemCF 实现细节**:(1) 共现统计:每用户点击序列(截断最近 2000 条控制 pair 量)内滑窗 w=10,对每个间隔 d∈[1,10] 向量化生成 (s[:-d], s[d:]) 对,scipy COO 累加后对称化;(2) 归一:sim(i,j) = cooc(i,j)/√(cnt_i·cnt_j)——余弦归一惩罚热门 item 的"与谁都共现",与双塔的 logQ 是同一思想(去流行度偏差)在不同方法里的体现;(3) 每行截断 Top-100 相似对,控制内存与噪声;(4) 推断:用户最近 50 个去重点击的相似度行向量求和,排除已点击,argpartition 取 TopK。1K test R@50=0.0047,是热门的 2.2 倍。

### 5.2 双塔:结构、损失与两次关键修复

**结构**:用户塔 = 最近 50 次点击的 item embedding mean-pooling(padding mask 后除以真实长度)⊕ 用户 ID embedding → MLP(128→128→64);物品塔 = item ⊕ 作者 embedding → 同构 MLP;两侧输出 L2 归一。embedding 均为 64 维,正态初始化 std=0.02。

**损失(in-batch sampled softmax)**:batch 内 B 个 (u,i⁺) 对,logits = U·Vᵀ/τ(B×B),对角线为正例做交叉熵,温度 τ=0.05。归一 + 温度的作用:余弦相似度值域窄,除以 0.05 等于放大 20 倍再 softmax,否则分布接近均匀、梯度微弱;温度实质控制对困难负例的关注度,是对比学习标配。

**修复一:logQ 采样偏差校正**。in-batch 负例按点击频率 Q(j) 出现,热门 item 被过度当成负例、系统性打压。理论上要优化全库 softmax,batch 内近似的 importance 校正是 logits_j ← s_uj − log Q(j):Q 大(热门)补分多,抵消过采样惩罚。实现即交叉熵前对每列减 `log(点击占比)`。效果:Pure val R@50 从 0.07 → 0.267(近 4 倍)——是"修 bug"级而非"调参"级的差异,也是我面试必讲的第一个坑。

**修复二:最优 checkpoint + 早停**。Pure 上 val R@100 在 ep2–3 见顶后持续回落,原实现保存末轮模型,test R@50 損失 15%(0.199 → 修复后 0.230)。改为每 epoch 评 val,R@100 创新高则深拷贝权重,patience=2 早停,test 用最优权重。1K 上双塔训到 ep8 才饱和(val R@100 0.0477→0.0515),早停机制统一处理了两个数据集完全不同的收敛节奏。

**评估实现**:全量候选 item 分块(8192)过物品塔得矩阵 V,评估用户分块(1024)过用户塔,U·Vᵀ 精确内积检索,屏蔽 OOV 桶与已点击。3 万候选无需 ANN;规模扩大换 faiss(HNSW 精确优先,亿级 IVF-PQ 换内存)。

**结果与解读**:1K test R@50=0.0100 / R@500=0.0599,是 ItemCF 的约 2 倍;Pure 上两者接近(0.230 vs 0.191)。解读:候选空间 7.5k→31.7k,记忆式共现的矩阵急剧稀疏,泛化式稠密检索优势随规模放大——这是"为什么工业界召回主力是双塔"的亲测注脚。

## 六、精排:DIN 基线

**结构**:target 表征 = item_h ⊕→加和← 作者 ⊕ tag 三个 64 维 embedding 之和;历史序列 = 最近 200 条点击的 item_h embedding;target attention:对每个历史位置拼接 [h, t, h⊙t, h−t](256 维)过 MLP(→64→1)出打分,padding 位置填 −1e9 后 softmax,全空历史用 `mask.any()` 强制输出零向量;加权和得兴趣向量 u;输出层输入 [t, u, t⊙u, user_emb](256 维)过 MLP(256→128→64→1)出 logit,BCE 训练。

**关键设计的答辩点**:(1) target attention 相对 mean-pooling 的本质是"按候选按需读取多峰兴趣"——候选是游戏时只激活历史中的游戏行为;(2) 显式拼 h⊙t、h−t 是帮小 MLP 免学乘性/差性交互,DIN 原文设计;(3) 我用 softmax 归一而原文特意不归一(保留兴趣强度)——工程取舍,训练更稳,强度信息由别处(GSU 检索量、类目频次)部分保留,被问必须能说出原文动机。

**训练**:Adam 1e-3,batch 1024,负例即曝光未点击。**One-epoch 现象**:val GAUC ep0=0.5825 → ep1=0.5731,第一个 epoch 即最优,ep1 开始过拟合——大规模稀疏 ID embedding 的参数只被自己的样本更新,第二遍开始记忆噪声。DIN/SIM/MMoE 在两个数据集上全部复现此现象,而双塔例外(负例组合逐 epoch 变化,等效数据增广)——两类模型训练动力学的对照是高频考点。每 epoch 用 val 抽样 30 万条评估以省时,test 全量。

**结果**:1K test AUC 0.7159 / GAUC 0.5769。单 epoch 训练在 4060 上约 8–11 分钟(8,647 steps),瓶颈在 DataLoader 的 per-sample searchsorted 而非 GPU。

## 七、SIM 长序列:三次迭代(项目方法论高光)

**目标**:把窗口外的远期行为用起来。范式是 SIM 的两阶段:GSU(General Search Unit)从全生命周期检索与 target 相关的行为子集,ESU(Exact Search Unit)对子集做精细 attention。

**v1:同类目最近 K=50 条**。TagHistory 以 (uid, tag) 为键预构建时间有序的行为数组,检索是 searchsorted 二分 O(log n);模型侧长短两支路各一套独立 TargetAttention,输出拼接 [t, u_short, u_long, t⊙u_short, t⊙u_long, user](384 维)进 MLP。**结果:GAUC 0.5747,与 DIN 逐位持平。** 诊断过程:先单测排除实现 bug;再做量化推理——日均点击 280 条,短窗口 200 条覆盖最近约 17 小时;高频类目的"最近 50 条同类目"时间跨度约 1–2 天,大部分落在短窗口内。**结论:长支路检索到的是短支路已有的信息,伪增量。**

**v2:不相交检索**。实现:短窗口函数 `window()` 额外返回窗口内最早一条的时间戳 t_start,GSU 的检索右边界从样本时刻 t 改为 t_start——两支路按构造在时间轴上零重叠,无需检索后去重,可单测。**结果:val GAUC 0.5851(+0.3pt,三组里唯一超过 DIN 的),test 0.5775 基本持平。** 解读:远期信息确有增量(val 可见),但被两个因素压制——检索粒度(66 个一级标签下"同类目"太宽)与长尾表征质量(检索出的多为哈希桶 ID)。这个诊断直接派生了后续两个方向:细粒度检索键(v3)与语义增强(第八章)。

**v3:二级类目检索键**。GSU 键从 tag1(66 类)换成 cat2(175 类),检索键可配置(`--set rank.gsu_key=cat2`),target 的 embedding 仍用 tag1(cat2 只管检索不进表征)。**结果:test GAUC 0.5976(在语义版基础上 +0.29pt)**——"影视"变成"电视剧/影视解说/…"后,检索子集与 target 的相关性质变。粒度的边界:三级类目(千级)会让每键行为过少、长支路频繁空转,最优粒度取决于行为密度;极限形态是 embedding 软检索(ETA 的 SimHash / TWIN 的 GPU 精确检索),我的语义向量是现成的检索资产,列为下一步。

**序列长度 scaling 消融**(SIM+sem):hist_len 50/100/200/400 → GAUC 0.5857 / 0.5894 / 0.5947 / 0.5946。单调上升、200 饱和。这条曲线是长序列叙事的定量地基:暴力加窗的边际收益归零(且注意力成本 O(L) 线性涨),而 v3 证明窗口之外仍有 +0.3pt 可拿——"检索式长序列"不是炫技,是唯一通路。

## 八、语义增强(最大单项涨点)

### 8.1 离线编码管道

文本源:caption + 封面文字拼接(如"#王者荣耀妲己秀 刺客法爆流妲己教程…"),空文本跳过。编码器 bge-small-zh-v1.5(中文检索任务预训练,512 维输出,L2 归一),batch 256 GPU 编码,覆盖 3,748,933 个视频(85.79%),437 万条约 40 分钟。中间产物是 fp16 memmap(437 万 × 512 ≈ 4.5GB,磁盘上),不进显存。

降维:IncrementalPCA 在至多 100 万条非零行上分批(8192)拟合,降到 64 维,保留方差 54.6%;全量分块 transform 后存 fp16 语义表 `sem_emb.npy`(437 万 × 64,560MB),行 0 与未覆盖行置零。**为什么 64 维**:512 维 fp16 表 4.5GB 超出 4060 常驻预算,64 维 560MB 可全程驻留 GPU;54.6% 方差看似丢一半,但语义主方向(话题/品类)集中于头部维度,尾部多为表述风格噪声——事后由 +1.7pt 的收益验证了取舍合理,128/256 维消融列为待办。

### 8.2 接入方式与"冻结"的理由

模型侧:语义表注册为不参与训练的 buffer,一层可训练 Linear(64→64) 投影后,**加**到 target 与长短序列每个 token 的 ID 表征上(`base_emb + proj(sem[vid])`)。三个决策的答辩:

- **冻结**:65 万训练点击 fine-tune 亿级参数编码器必然灾难性遗忘 + 过拟合;冻结表 + 轻量投影让梯度只学"语义空间→行为空间的映射",完整保留世界知识。这是小数据接大模型的标准姿势。
- **加法而非 concat/gating**:加法不改变下游一切结构的输入维度(attention/MLP 全复用),参数最少;且允许残差式分工——行为充足的 item 由 ID embedding 主导,冷 item 退化为纯语义。gating 是验证价值后的升级项,不是起点。
- **双 ID 机制**在此闭环:序列 token 的可训练表征查 `iid_h`(哈希),语义查 `vid`(稠密),长尾行为第一次同时拥有"可学的粗身份"和"精确的内容身份"。

### 8.3 三重验证

- **2×2 消融**:语义给 DIN +1.74pt、给 SIM +1.72pt(GAUC),增益不依赖长序列结构,是独立正交的贡献。
- **热度分桶**(核心证据):±sem 的 test AUC 差按 item 训练期曝光分桶——0 次 +1.64、1–9 次 +1.40、10–29 +0.82、30–99 +0.36、100+ +0.51。增益随冷度单调放大,证明语义替代的正是缺失的协同信号;100+ 桶仍 +0.5pt 说明热门侧也受益于"用户历史 token 的语义化"。
- **无偏复评**:随机流量上增益完整保留(+1.7pt GAUC),排除"更像旧策略"的解释。

**Case study 佐证**(可讲的例子):语义近邻里同系列视频 cos 0.94、"斌娜夫妇"情侣号内容自动成簇;"救回案例"——用户近期点娱乐综艺,目标视频《78 岁老奶奶身材完爆杨幂 #达人秀》被无语义模型排在 11% 分位、语义模型排到 93% 分位且实际被点击。

## 九、多任务:MMoE

**结构**:SIM 的整个表征层(双 attention + 语义)为共享底座,输出 384 维特征 x;4 个专家 MLP(384→256→128→64→64);每任务一个 Linear(384→4) 门控 softmax 加权专家输出;每任务独立塔(64→64→1)。任务 = click / long_view / like,等权 BCE(三 logit 与三标签的逐元素 BCE 取均值),早停依据主任务 click 的 val GAUC。

**结果**(test):click 0.7302/0.5939(与单任务 0.5947 差异在噪声内,无损)、long_view 0.7419/0.6152、like 0.9180/0.5961。解读:4 专家容量下三任务基本不冲突(click 与 long_view 本就高相关);like 的 AUC/GAUC 剪刀差是 GAUC 必要性的最佳教材。等权的合理性:三个 BCE 同量纲,like 因正例率 1.5% 其 loss 绝对值天然小,已是事实上的弱任务;GradNorm/不确定性加权是"等权失效有证据后"的升级项。

**融合权衡工具**:对 test 每用户的曝光按 score=(1−α)p_click+α·p_lv 重排,取 top-10 计算点击命中、长观命中、类目多样性,α∈{0,0.2,…,1.0}。结果:α 0→0.4 点击 0.6237→0.6231(−0.06pt,无损)而长观 0.5116→0.5323(+2.1pt)——免费午餐区;α→1.0 再换 3.5pt 长观需付 1pt 点击;多样性 4.94→4.69 单调收窄。产出物是一条实测 Pareto 曲线——"融合权重是业务决策,算法负责把曲线摆上桌"。

## 十、生成式预训练(负结果及其价值)

**方案**:2 层 TransformerEncoder(d=64,2 heads,FFN 256,pre-norm),输入最近 50 条点击的 item_h embedding + 可学位置编码,取最后一个非 padding 位置的状态 L2 归一后,与 target item embedding 做 in-batch softmax(τ=0.05,带 logQ 修正)——即自回归 next-item prediction。在 642,746 条点击事件上训 2 epoch(每 epoch 仅 35 秒),导出 item embedding 表初始化 SIM 微调。

**结果:GAUC 0.5945 ≈ 基线 0.5947,无增益。** 归因(比结果重要):(1) 语料严重不足——64 万事件对 23 万词表,平均每 token 不到 3 次出现,预训练 loss 停在 4.83 远未收敛;(2) 信号同源——预训练目标(点击流的下一跳)与下游 CTR 提取的是同一信息源,增量先天有限;(3) CTR 微调的 one-epoch 特性让初始化的影响被快速覆盖。**可迁移的判断**:预训练在推荐中的价值依赖数据规模与信号异源性(跨域行为、搜索日志、内容模态),同源小数据上做预训练是负性价比——这与工业界经验一致,负结果由此变成了"知道边界"的证据。

## 十一、工程实现与可复现性

**资源账**(RTX 4060, 8GB):最大参数块 iid_h embedding 231,700×64 ≈ 1,480 万参数(59MB,加 Adam 一二阶矩 ×3 ≈ 180MB);语义表 560MB fp16 buffer(不占优化器);序列激活(batch 1024 × 250 token × 64 维及 attention 中间量)数百 MB;总显存占用 2–3GB,留有加倍余量。排序单 epoch 8–11 分钟、双塔 1K 单 epoch 约 1 分钟、编码 40 分钟、全项目 GPU 总时长约 20 小时。

**性能瓶颈与对策**:训练瓶颈在数据侧——每样本 2 次 searchsorted + numpy 切片,GPU 不满载;对策:历史索引预构建为 per-user(或 per-(user,cat))连续 numpy 数组、4 worker 并行、避免 pandas 逐行操作。进一步优化(未做):离线物化定长历史到 parquet,训练零计算换磁盘。

**质量保障**:关键正确性单元测试(泄漏、隔离、截断、双 ID 一致);合成数据生成器复刻真实 schema,任何改动先冒烟后上真;配置驱动 + 结果流水账(results.csv);语义表与 video_map 的确定性构建保证 preprocess 重跑不破坏已编码资产。

**踩过的值得讲的坑**:logQ 缺失(4 倍差距)、末轮保存(−15%)、SIM v1 伪增量(重叠检索)、统计特征的时间泄漏(主动弃用)、Windows/WSL 环境链(venv 放 Linux 侧、驱动与 CUDA 版本匹配)——每个坑都有"现象→诊断→修复→验证"四步,面试讲坑比讲成功更立体。

---

## 十二、局限与演进方向(被问"接下来做什么"的标准答案)

1. **检索升级**:GSU 从离散类目键升级为 embedding 软检索(语义向量现成),对标 ETA(SimHash 近似)/TWIN(精确并行);预期把长序列增量从 +0.3pt 再往上推。
2. **语义侧**:PCA 维度消融(128/256)、缺失 caption 的可学习默认向量、多模态(封面图过 Chinese-CLIP 与文本对齐)、RQ-VAE 量化成语义 ID 接生成式检索(TIGER 路线)。
3. **范式演进**:统一 token 化已就位(item/作者/类目/语义同空间),放大路径是自回归 Transformer 编码全行为流 + next-token 目标(HSTU,工业已证明 scaling law 存在)、sparse MoE 承接用户异质性(现有 MMoE 门控改 Top-k 路由即得)、随机曝光数据做 IPS/DR 的 off-policy 评估进而接离线 RL——每一步都从现有资产出发,不是空谈。
4. **诚实的边界**:1000 用户使 GAUC 方差偏大,小于 0.3pt 的差异不应过度解读;曝光空间训练带来的 SSB 未处理;数字不可直接外推到工业规模——结论的适用范围本身是结论的一部分。
